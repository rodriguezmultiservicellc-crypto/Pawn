  KEY DECISIONS LOCKED THIS SESSION:
    - LOAN-RATE MIN APPLIES PER-DAY, NOT PER-MONTH. interestAccrued
      floors `daily = MAX(principal * rate / 30, min / 30)` so a
      partial-month redemption pro-rates cleanly. A $10 loan with
      10%/mo + $20 min charges $0.67/day; 30 days = $20, 15 days =
      $10. Defensive: works even if rate=0 with min set (the form
      won't allow that today, but math handles it).
    - PER-RATE MIN, NOT TENANT-WIDE MIN. Each tenant_loan_rates row
      gets its own optional min_monthly_charge so an operator can
      configure "10%/mo, $20 min" and "5%/mo, $10 min" as two
      different rate rows. More flexible than one tenant-wide knob.
    - MIN-CHARGE SNAPSHOTTED ONTO loans AT INTAKE. Same pattern as
      interest_rate_monthly: the value copies from the chosen rate
      to loans.min_monthly_charge at /pawn/new submission, frozen
      for the life of the loan. Future rate-menu edits never alter
      existing loans.
    - TENANT-WIDE MIN LOAN AMOUNT LIVES ON settings. /pawn/new
      server action validates `principal >= settings.min_loan_amount`
      after schema parse but before any DB writes, returns
      'principal_below_tenant_min:X.XX' field error.
    - MIN-CHARGE THREADED THROUGH ALL 6 payoffFromLoan CALLERS
      (staff detail page, staff actions, portal list, portal detail,
      portal pay-by-link action, cron maturity reminders, Stripe
      webhook). Critical: Stripe webhook computes payoff for
      automated redemption flips — under-billing here would write
      'redeemed' status on insufficient funds.
    - REPAIR WORKFLOW: 'assigned' SITS BETWEEN awaiting_approval AND
      in_progress. Operator routes ticket → assigned (with assigned_to
      + assigned_at). Tech claims → in_progress (with claimed_at).
      Legacy bypass (awaiting_approval → in_progress directly) kept
      for operators who don't use the assignment flow.
    - REPAIR WORKFLOW: 'tech_qa' SITS BETWEEN in_progress AND ready.
      Tech finishes hands-on work → tech_qa. QA approve → ready. QA
      fail → back to in_progress. Legacy bypass (in_progress →
      ready directly via markCompleteAction) kept.
    - AUTO-TIMER HOOKS DRIVEN BY workflow.ts HELPERS.
      shouldOpenTimerOnEnter('in_progress') = true and
      shouldStopTimerOnLeave('in_progress') = true. claim /
      parts_received / qa_returned auto-OPEN a timer for the assigned
      tech; needs_parts / sendToQa / markComplete / markAbandoned /
      void auto-STOP all running timers. Idempotent helpers
      (openTimerForTech / stopRunningTimers) — no error if a timer
      is already open or no timer is running.
    - MANUAL PAUSE/RESUME PRESERVED VIA existing startTimer /
      stopTimer (e.g. lunch breaks). The auto path doesn't replace
      the manual path; both coexist.
    - assignTechnicianAction NOW MANAGER-ROLE-ONLY. Was open to all
      STAFF_REPAIR_ROLES (incl. pawn_clerk + repair_tech). Now
      MANAGER_ROLES only (owner/manager/chain_admin). Routing work
      is a managerial decision; techs claim work, they don't move
      tickets between jewelers.
    - CLAIM-TICKET AUTHORIZATION: must be the assigned_to tech, OR
      a manager-level role (owner/manager/chain_admin). Manager
      override lets owners reassign-and-claim in one step if a
      jeweler is out. If a manager claims an unassigned ticket,
      assigned_to gets set to the claiming user so the timer + audit
      trail line up.
    - TECH-INBOX AUTO-FILTER on /repair list: when role=
      'repair_tech' and no explicit assignedTo param and no
      ?showAll=1 flag, default assignedToFilter to ctx.userId. So
      jewelers land on their own queue by default; manager+ roles
      see everything.
    - assigned_at STAMPED ON FIRST ROUTING ONLY. Re-assignment keeps
      the original timestamp so we can measure queue-to-claim time
      across re-routings.
    - STATUS BADGES: 'assigned' uses cool rausch-deep tone (queued);
      'tech_qa' uses muted success tone (final lap). Status enums
      now have 11 values total (was 9).
