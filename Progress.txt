    - PORTAL INVITE FLOW. Operator at /customers/[id] → "Send portal
      invite" → server creates customer_portal_invites row (UUID

      bound → user_tenants(role='client') upsert + customers.
      auth_user_id link + invite consumed → set pawn-active-tenant
      cookie → redirect /portal/loans.
    - PORTAL CLAIM ROUTING. /portal/claim/* lives at src/app/portal/
      claim/<token>/ NOT inside (portal) route group. Reason: the
      (portal) layout enforces tenantRole='client' which would
      deadlock the brand-new auth user. Proxy.isPortalPath has a
      carve-out for /portal/claim AND /portal/login.
    - PORTAL LOGIN IS DEDICATED. Customers go to /portal/login (NOT
      staff /login). Magic-link request via per-tenant Resend; same
      pattern as invite. After magic-link verification, /api/portal/
      sign-in-bridge sets the active-tenant cookie BEFORE the
      (portal) layout runs (otherwise the layout's tenantRole check
      would fail and bounce to /no-tenant).
    - OPTIONAL PASSWORD. Customers can set one from /portal/account
      via supabase.auth.updateUser({password}). Portal login form has
      a single smart "Sign in" button: password if filled (uses
      signInWithPassword + nav to sign-in bridge), magic-link if
      empty. Forgot-password is "leave blank, get magic link" — no
      separate flow needed.
    - SUPABASE AUTH URL CONFIG. Defaults are localhost:3000 site_url
      + empty uri_allow_list. Patched live to pawn-three.vercel.app.
      scripts/{inspect-auth-config,set-auth-urls}.mjs check this in
      so future drift is one command away. Without these, magic
      links redirect to whatever Supabase's site_url is — and any
      redirect_to we pass that isn't on the allow list gets ignored.
    - generateLink TYPE FALLBACK. type='invite' fails when the auth
      user already exists (re-invites, multi-tenant customers). Auto-
      retry with type='magiclink' when the error message contains
      "already" / "exists" / "registered".
    - APP URL DERIVATION ORDER. (1) NEXT_PUBLIC_APP_URL if set AND
      not localhost; (2) request headers (x-forwarded-host +
      x-forwarded-proto, Vercel auto-injects); (3) VERCEL_URL; (4)
      NEXT_PUBLIC_APP_URL even if localhost. Means the magic-link
      flow works without any env-var setup on the deployed URL —
      but the operator should still set NEXT_PUBLIC_APP_URL for
      Stripe / eBay / dunning paths that read it directly.
