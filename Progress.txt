  KEY DECISIONS LOCKED THIS SESSION (Session 23):
    - HYBRID API KEY MODEL. Platform-wide
      GOOGLE_PLACES_API_KEY env var as default;
      per-tenant override in
      settings.google_places_api_key (nullable).
      cache.ts resolves at the call site
      (per-tenant first, env fallback). Sophisticated
      tenants can self-host; default tenants use
      the platform key. Mirrors the Resend
      precedent (env fallback + per-tenant
      override).
    - SILENT FAIL-SOFT (Decision 4). Public widget
      renders iff place_id set AND last fetch
      succeeded AND ≥1 review survives min-star
      filter. All errors surface only in /settings,
      never on the public landing. Encoded in
      isWidgetRenderable + the four null gates of
      loadPublicReviews.
    - SINGLE-WRITER RULE for tenant_google_reviews.
      Every INSERT/UPDATE to the cache table flows
      through src/lib/google-reviews/cache.ts.
      Test-connection action calls refreshReviews,
      not direct DB writes. Settings-clear path
      DELETEs cache row but that's not a write
      to cache business logic — it's data hygiene.
    - PLACEHOLDER-ROW PATTERN for first-failure
      visibility. writeError UPSERTs a row with
      payload:{} so /settings shows "Last sync
      failed: <error>" before any successful fetch
      has happened. Without it tenants with
      typo'd place_id would show "Pending"
      forever.
    - NEXT.JS after() FOR SWR RELIABILITY. Stale-
      while-revalidate path uses
      `after(() => refreshReviews(tenantId))` from
      'next/server' (NOT bare void). Vercel
      runtime can reclaim the function before fire-
      and-forget promises resolve; after() registers
      with waitUntil so the background refresh
      reliably completes after the response is
      sent. Subagent code-quality review caught
      this.
    - HARDCODED ENGLISH IN STAFF SETTINGS UI.
      Plan over-applied i18n to the operator
      settings page. Project's actual pattern is
      staff settings hardcode English (operators
      mostly English-speaking — see /settings/
      loyalty/content.tsx + /settings/integrations/
      content.tsx). Only customer-facing surfaces
      use i18n. Plan Tasks 7 + 8 + 12 updated
      inline to reflect this; landing.reviews.* is
      the only i18n block this slice adds.
    - LOADPUBLICREVIEWS REFACTORED TO SINGLE-ARG.
      Plan threaded google_place_id +
      google_reviews_min_star_floor through
      PublicTenant, but those columns live on
      `settings`, not `tenants`. fetchPublicTenant
      reads tenants only — tsc rejected the column
      additions. Refactor: loadPublicReviews now
      takes just tenantId; it reads
      settings.google_reviews_min_star_floor itself
      via admin client in parallel with the cache
      row. Admin client is safe at this call site
      because the tenantId is already validated as
      a published tenant by fetchPublicTenant
      upstream.
    - WRITEERROR INSERT BRANCH SURFACES NON-PK
      ERRORS. The race window between SELECT and
      INSERT could let a concurrent success-UPSERT
      cause a 23505 PK collision in the error
      INSERT path. 23505 is benign + silently
      swallowed. Other errors now console.error.
      Subagent code-quality review caught this.
    - STAR ICON COLOR (Decision 8). Phosphor Star
      weight=fill, text-ink filled / text-hairline
      empty. NOT Warning Amber — that's a semantic-
      only token; using it decoratively violates
      the locked design system. Matches Airbnb's
      parent treatment of dark-on-cream stars.
    - SUBAGENT-DRIVEN-DEVELOPMENT FOURTH RUN
      VALIDATED. Same pattern as Session 22.
      Code-review caught two real bugs: (1) the
      writeError race-log gap, (2) the Vercel
      after() vs bare void. Plan-code itself had
      one bug caught by the implementer self-
      review (inverted truncateExcerpt assertion).
      Three real catches in 13 tasks — process
      continues to earn its keep on subtle
      tasks (cache.ts) but is overhead-heavy on
      mechanical pastes (client.ts, types.ts) —
      consider tagging subagent-driven only on
      load-bearing tasks in future slices.
