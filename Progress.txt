    - PHASE 9 (A) ADMIN NOTIFY DETECTS "FIRST SIGNUP" via
      pre-upsert SELECT of tenant_subscriptions: no row OR

      dedicated 'saas_admin_new_tenant' kind yet (future migration).
    - PHASE 9 (A) PLAN-CHANGE WRITES TO STRIPE DIRECTLY (no Checkout
      redirect for existing subscribers). Endpoint POST /api/stripe/
      saas/change-plan: retrieveSubscription → grab item.id →
      PATCH /subscriptions/{id} with items[0][price]=newPriceId +
      proration_behavior=create_prorations. Idempotency-Key bound to
      (tenant, plan, cycle, sub) tuple prevents double-billing on
      button-mash. The customer.subscription.updated webhook
      reconciles the local DB. Refuses gracefully if no active sub —
      caller should fall back to /checkout.
    - PHASE 9 (A) BUTTON LABEL IS CONTEXT-AWARE. /billing PlanCard
      shows "Current plan" (disabled) for exact match, "Switch
      cycle" for same-plan-different-cycle, "Switch to this plan"
      for different-plan-with-active-sub, "Choose" for no-sub. Stops
      operators from accidentally double-paying for the plan they
      already have.
    - PHASE 9 (B) INLINE CALCULATOR DOESN'T LIFT COLLATERAL STATE.
      InlinePawnCalculator reads the parent form via document.
      getElementById(formId) + new FormData(form). Keeps Collateral
      ItemsList's internal state untouched and avoids regressing
      the Session-8 form-reset pattern. "Use this amount" writes
      to the principal <input> via DOM mutation — uncontrolled
      input, no React-state surgery needed.
    - PHASE 9 (B) WATCH MODELS ARE PLATFORM-LEVEL (NO tenant_id) in
      v1. Every shop sees the same catalog. Per-tenant overrides
      can land later via watch_model_overrides, mirroring the
      spot_price_overrides pattern. RLS allows any authenticated
      user to SELECT (this is reference data, not customer data);
      writes go through admin client gated by requireSuperAdmin().
    - PHASE 9 (B) DIAMOND CERT IS A SCAFFOLD — provider routing
      done, bodies are TODO. Locks the call signature so the swap
      is one file when the operator obtains paid keys (IDEX or
      Rapaport). v1 always returns 'not_configured'; the existing
      Verify-on-LAB deep-link in StoneEditor remains the manual
      fallback.
    - VARIABLE-RATE MENU IS A NEW MODULE (not Phase 9 closure).
      Eddy explicitly called it out as separate. Per-tenant
      tenant_loan_rates table — not platform-level — because
      different shops use different rate structures.
    - VARIABLE-RATE FORM USES A HIDDEN FIELD. The visible <select>
      drives a hidden <input name="interest_rate_monthly"> that
      carries the actual decimal value to the server action. The
      existing Zod schema (z.coerce.number().min(0).max(0.25))
      accepts both preset and custom values without changes — no
      migration to loans table needed.
    - VARIABLE-RATE GRACEFUL FALLBACK. When a tenant has zero
      configured rates, the dropdown is hidden and a custom-only
      number input renders (with a help note pointing to settings).
      The migration seeds every existing tenant with a 10%/mo
      default so this fallback is rarely hit, but it exists for
      tenants between migration and first config save.
    - VARIABLE-RATE SINGLE-DEFAULT INVARIANT enforced two ways:
      partial unique index uq_tenant_loan_rates_default at the DB
      level, plus server-side "clear all others" UPDATE before
      saving the new default-flagged row. Belt + suspenders so
      neither path ever produces zero or two defaults.
    - NUMERIC COLUMNS STILL NEED RUNTIME COERCION. Both
      watch_models.est_value_min/max and tenant_loan_rates.

      just the same long-standing NUMERIC-vs-types divergence
      noted in Session 10's CLAUDE.md gotchas. Backlog item:
