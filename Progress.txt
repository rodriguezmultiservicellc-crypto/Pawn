    - Repair = stone setting: same table, service_type ENUM. Avoids
      duplicating the lifecycle / photo trail / time logs across two
      modules that diverge only at the mounting-type field.
    - Police-report format per-tenant: tenants.police_report_format
      ENUM defaulting to 'fl_leadsonline'. Adding new states = adding
      a new enum value + a new exporter file under src/lib/compliance/
      police-report/formats/. The dispatcher at
      src/lib/compliance/police-report/index.ts reads the enum and
      routes — never a fork.
    - Stripe Connect both surfaces: same per-tenant connected account
      drives Terminal (card-present in POS) AND Payment Links
      (customer portal payoff). One OAuth flow, one webhook endpoint,
      both surfaces.
    - compliance_log is its own table, write-once. Police-report
      exporter reads compliance_log — NOT loans / sales — so reports
      stay deterministic across edits / voids / corrections.
    - Customer ID retention enforced at the customer-record DELETE
      gate. FL = 2 years post-redemption / forfeiture (verify current
      statute before going live in any state).
    - Pawn ticket immutability after print: ticket print stamps a
      printed_at timestamp; trigger rejects core-field UPDATE on
      printed tickets. Corrections = void + re-issue.

  KNOWN GOTCHAS TO AVOID FROM DAY 1 (carried from Luna Azul + Abacus):
    - Next 16 file is `proxy.ts`, exported function named `proxy`.
      NOT middleware.ts / middleware.
    - Phosphor icons crash server components — only import in
      'use client' files. Use HTML entities or unicode glyphs in
      server-component pages.
    - useSearchParams() must be wrapped in <Suspense> for static
      gen — required by Next 16 build phase.
    - <form action={fn}> never has explicit encType. React 19 +
      Next 16 auto-generate it; explicit value causes hydration
      mismatch.
    - Storage uploads from server actions must use the admin
      (service-role) client — buckets default to no RLS policies,
      so the user-scoped client gets RLS-rejected.
    - npm run db:types via the safe wrapper. Never raw `>` redirect.
    - Service role key NEVER goes in NEXT_PUBLIC_*. Never in
      client code. Anon key is fine to expose.
    - DATABASE_URL via Supavisor pooler URL
      (postgres.<ref>@aws-1-us-east-1.pooler.supabase.com:6543) —
      NOT direct DB host (IPv6-only on standard tier).
    - GitHub commits MUST be authored by the identity Vercel has
      linked. Use `git -c user.name=... -c user.email=...` for
      inline override if local repo identity diverges.
    - Auth callback needs both PKCE (?code= in query) AND implicit
      flow (#access_token= in hash) handling. Implicit fragment
      can't be read server-side — needs a client-side handler at
      /auth/callback.
    - Portal invites: don't use Supabase built-in SMTP (rate-
      limited / shared sender). Use admin.generateLink({type:
      'invite'}) + email the action_link via the tenant's Resend.
    - RLS-helper-function pattern: my_*() functions are
      SECURITY DEFINER. Stable. Used by every policy.
    - profiles.role is GLOBAL only ('superadmin' | NULL). Never
      add per-tenant role to profiles. Never check
      profiles.role === 'owner'.
    - user_tenants has no FK to profiles — any embedded
      profiles(...) select returns null. Do a separate profiles
      lookup keyed by user_id.
    - Never taskkill /F /IM node.exe — kills sibling apps. Use PIDs.
    - Reset DB password = update DATABASE_URL in BOTH .env.local
      AND Vercel env. Audit script's URL parser must rsplit on the
      LAST '@' so passwords containing '@' don't break parsing.
