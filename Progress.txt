  KEY DECISIONS LOCKED THIS SESSION:
    - SUBDOMAIN + PATH FORM SHARE A SINGLE RSC.
      acme.basedomain.com/ rewrites to /s/acme via the
      proxy; basedomain.com/s/acme works directly.
      Same RSC handles both — landing rendering doesn't
      branch on URL form. Path form works the day the
      code ships; subdomain form lights up when
      wildcard DNS is configured. Avoids
      "subdomain-only" coupling that would block local
      dev and Vercel preview URLs.
    - PUBLIC RLS GATES ON THREE BOOLEANS (not just one).
      The tenants_public_landing_select policy requires
      public_landing_enabled = TRUE AND public_slug IS
      NOT NULL AND is_active = TRUE. Suspending a
      tenant via is_active=FALSE auto-takes their
      landing offline without a separate "unpublish"
      step. Reserving a slug without publishing is
      legitimate (typo-squat protection) — the slug+
      flag separation supports it.
    - RESERVED SUBDOMAINS LIVE IN APP CODE, NOT THE DB
      REGEX. The CHECK constraint enforces shape +
      length only. RESERVED_SUBDOMAINS list in
      tenant-resolver.ts gates writes (via the action
      pre-check) and reads (via the proxy). New
      reserved names — when we add platform infra
      subdomains — go in app code without a migration.
    - LANDING PAGE FETCH USES ANON SSR CLIENT, NOT
      ADMIN. Service-role bypasses RLS entirely; the
      whole point of the RLS policy is to make the
      public-read surface opt-in per tenant. Using the
      anon client means the policy is the gate, not
      the SELECT list. PUBLIC_TENANT_COLUMNS still
      narrows to "what's safe to ship" as defense in
      depth.
    - I18NPROVIDER `persistRemote` PROP, NOT A FORK.
      Two layouts (staff/portal) need profile sync; one
      (public) doesn't. Forking I18nProvider would
      duplicate the language-state machinery; a
      persistRemote=false flag is one extra line and
      lets the same provider serve both. Default
      preserves staff/portal behavior.
    - HOURS PAYLOAD VALIDATION IS IMPERATIVE, NOT ZOD.
      buildHoursPayload runs on 21 form fields with
      cross-day rules (both open + close set or both
      blank, open<close, regex on each). Expressing
      that in Zod would need 7 refines + per-day
      objects; an imperative loop reads cleaner and
      surfaces day-specific error messages. Schema
      shape (`{open, close, closed}`) lives at the
      runtime boundary in normalizePublicHours, NOT
      in the DB — JSONB column is permissive on
      purpose so schema evolutions (multi-interval
      days, holiday overrides) don't need migrations.
    - NEXT_PUBLIC_BASE_DOMAIN IS OPTIONAL, NOT
      LOAD-BEARING. Local dev sets it empty; the
      proxy short-circuits the subdomain branch and
      everything still works via /s/<slug>. Vercel
      production sets it once wildcard DNS is
      configured. Avoids forcing the env var into the
      Phase 0 punch list as a gating item.
    - DROPPED THE SPLIT-WRITE PATTERN POST-DB:TYPES.
      Sessions 17 and 18 used split UPDATEs with the
      second one cast `as never` because the new
      columns weren't typed. Once db:types regenerates
      with all the new columns, the second update
      rejoins the first — fewer round-trips, simpler
      audit-log entry, no `as never`. Keep the
      pattern in mind for the NEXT migration's first
      session, then collapse on db:types regen.
